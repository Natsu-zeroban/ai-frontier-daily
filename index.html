Claude Code 是怎么设计出来的：下一代设计师负责什么

观点·深度·Dive Club × Meaghan Choi（Claude Code 设计负责人）·晚点再听 LaterCast·~4 min·原文
一句话：命令行没有界面，反而更需要设计。当"做出来"变得很容易，设计师的价值从画布转向 discernment（可执行的取舍判断）——分辨此刻该塑形还是该执行、什么值得做、什么该克制。

没有界面，反而更需要设计

心智模型、交互模型、状态如何被感知，都是设计，只是没表现为图形界面。终端内容一起向上滚、没有渐进披露、开发者反而期待更高信息密度——每一样都要重新设计。连"计划模式 / 自动模式"的快捷键切换都吵了很久：要够快、不与终端已有快捷键冲突、还要让你随时知道自己在哪个模式。

先设计"它真的懂我"的魔法瞬间

用户最大的认知跨越不是接受聊天，而是相信 Claude 能读取整个代码库、直接写回文件。团队因此把注意力压在"第一次感受到魔法"上：全屏后的橙色区域、ASCII 吉祥物、动画，给冷峻终端加一点温度。信任来自两个信号——它确实理解当前环境，也会清楚告诉你它正在做什么。

好产品先长在工作流里

Artifacts 的起点不是产品蓝图，而是大家开始"互发 HTML 文件"这个笨拙动作暴露了真实需求。团队顺着使用过程补齐能力（生成→托管→分享→迭代→固定版本），最后才处理命名。

"当你注意到工具本身时，就是我们失败的时刻。它应该彻底让开，不挡住你的工作。"

Figma 没消失，交付边界变了

她先用 HTML 快速迭代，确认形态后再让 Claude 按真实组件重建成可提交的 PR——设计稿、可交互原型、代码不再有固定交接线，而是同一对象逐步提高保真度。选工具的标准很简单：此刻哪种更快、更清楚，就用哪种。

AI 从个人工具变成组织成员（Claude Tag）

不是和许多个 Claude 工作，而是整个组织和同一个 Claude 工作——它同时参与数千段对话、共享知识记忆、以组织成员身份拥有 GitHub / Google 权限。她很多 PR 直接从 Slack 一句话发起："去掉阴影"，Claude 自己找页面、开分支、建草稿 PR、附前后截图、等 CI 通过、请工程师审查。

可以外包执行，不能外包判断

新的关键能力是判断自己处于"塑形"还是"执行"阶段：想法没成立时和它讨论结构与必要性，方向明确后再让它加速。如果产品形态本身是错的，越快、越精致的执行只会更快抵达错误答案。测试版可以很难看——它验证的是底层心智模型是否成立，而不是精致度。

"能做出一切，不代表一切都应该被做出来。"未来设计师的三个特征：在混乱中保持好奇并持续更新判断、分辨产品塑形与执行打磨、对从想法到发布的完整闭环负责。

把 Skill 做稳：我开源了 Skill Engineering 1.0

干货·工具·艾笑 AI·~5 min·原文
一句话：一个开源工具，管一个 Skill 的完整生命周期——先判断值不值得做，再创建、检查、改进、验证，改正式版前一定先让你确认。它交付的不是一段长 Prompt，而是"边界清楚、能真实用、以后不会越改越乱"的能力。

核心工作流（以"会议纪要 Skill"为例）

  1. 不立即写文件：先确认几个真正会改变结果的问题——输入是什么、输出存哪、是否要跨项目复用、哪些相似请求不该触发、原文缺负责人/截止时间时怎么办。
  2. 写入前先给完整方案：应该触发 / 不应该触发 / 输出规则 / 第一版边界，你回复"可以写入"才创建。
  3. 改 bug 不堆补丁：真实反馈来了，先判断问题在哪一层→创建独立候选不覆盖正在用的版本→加一个回归场景防复发→你看差异再决定是否应用。
关键理念：一个 Skill 是否符合工程架构，首先不是看目录有多复杂，而是看职责、触发边界、停止条件、副作用是否在写入前就说清楚了。

四个用例，验证四种能力

  • 会议纪要 → 轻量创建：简单需求保持轻量，缺失字段标"待确认"不猜测。
  • GitHub Issue 分类 → 外部只读边界：第一版严格只读，即使你临时让它顺手写回 GitHub，也停在已批准的边界外。
  • Web 应用验收 → Skill 调脚本：自然语言触发和安全边界放 Skill 里，登录/搜索/导出/断言交给浏览器脚本，产出截图+CSV+报告。
  • 研究证据包 → 复杂架构治理：多文件结构，入口/契约/输出规范/回归用例分开管，遇高风险外部发布先停。
GET STARTED（Codex）
npx skills add https://github.com/wukongai/skill-engineering 然后对 Agent 说："用 Skill Engineering 帮我做一个 Skill，先判断是否真的需要；如果需要，让我先看到职责边界和创建方案，我确认后再写入。"
作者的观点：现在最该学的是 Skill。模型和工具由厂商提供，但把它们接进你工作流的连接层是 Skill——学 Skill 不能停在"安装"，要能读懂它在做什么、判断是否适配、改成自己的版本、并长期保持不漂移可验证。

一个人如何管理几十个 AI 程序员？

干货·方法论·刘小排·~4 min·原文
一句话：Peter Steinberger 一个人、五台机器、几十个 AI 任务并行——靠的是 Git 的老功能 Worktree。分支管"路线"，Worktree 管"现场"；一句话：分支是排班的，Worktree 是发工位的。

分支 vs Worktree

分支=代码往哪个方向走（主干是国道，分支是岔路，最后汇合）。Worktree=同时打开几个代码现场（同一仓库在硬盘上铺开成多个独立目录，这个改登录、那个修支付，互不干扰）。AI 时代门槛大降：以前要 stash→切分支→改→切回，任务一多脑子和代码一起乱；现在几条路线各有现场同时开工，你只需用大白话让 Claude Code / Codex 启用。

什么时候用谁：五种情况（从简到繁）

  1. 直接在 main 改：改文案、调价、修 typo 这种五分钟小事，开分支都浪费。
  2. 只开分支：大改动、周期长、可能推倒重来（重构鉴权、迁数据库）——一个人串行做，一个目录就够。
  3. 同一分支开多个 Worktree：路线只一条但有并行子任务，比如一个支付分支下同时接 Stripe / Paddle / PayPal 五个 provider，五个 Agent 并行做完一起合回。
  4. 分支 + Worktree 一起上：多条路线同时进行（落地页 / 账单 bug / 接新模型各开一分支、各配一现场）。
  5. 全矩阵：多分支 × 每分支多 Worktree × 多台机器——Peter 模式。判断标准就一条：你的任务是几件、几层？

给独立开发者的三条启发

  • 练的第一项能力是拆任务，不是写代码。拆得好，五个 Agent 一晚顶一周；拆不好，五个 Agent 互改对方代码，一晚上收拾残局。
  • 基础设施要先于功能投入。Peter 那张截图里一大半任务是删死代码、修测试、修 CI——顶级开发者派 AI 的大头是维护地基。地基越稳，敢同时派的 Agent 越多，这才是真杠杆。
  • 当产品经理，别当码农。当"手"变成几十个 Agent，最稀缺的不再是时间而是判断力：做什么、不做什么、哪个优先、交上来的代码合不合。
以前一个程序员一个工位（碳基工位）；现在一个程序员带几十个 AI，每个都得配一个工位（硅基工位）。

Netflix 产品与技术负责人：AI 时代为什么更需要系统型人才

观点·Lenny's Podcast × Elizabeth Stone（Netflix 首席产品与技术官）·晚点再听 LaterCast·~4 min·原文
一句话：当 PM 能写代码、设计师能做原型、工程师能碰产品，组织靠什么接住突然多出来的产出？Netflix 的答案：专业能力依然值钱，但更愿意把机会给"能看见整套系统"的人。
  • 岗位边界不会消失，但进入磨合期：支持大胆试，但"能做 ≠ 可以上线"——原型进生产仍要回答扩展性、测试、风险。AI 让每个人多会了几种语言，却没自动补上判断力。
  • 原型多 ≠ 走得更快：几千个原型若不指向重要业务问题，只是散落的半成品。速度要用"更早得到可验证的假设"来衡量，不是数你做了多少东西。
  • 为什么开始抢系统型人才：Agent 跨系统工作，每个都要知道哪份数据是真的、能访问什么、哪些操作要拦住。需要能看遍所有业务域、再抽象成可复用基础构件的人——好平台先把团队送到 80%，把护栏直接写进"铺好的路"里。
  • 设计师价值从页面移到整套体验：最怕各团队各自借 AI 提速，拼成一个"弗兰肯斯坦"——局部都能用，放进完整体验里彼此打架。设计师要把"什么算好"讲清楚、做成可复用的判断。

怎么练"系统思维"——方法很小

  • 把问题往外退一格：拿到任务先问它解决什么消费者问题、能不能跨类型扩展、值不值得沉淀为平台能力。不用顺手解决全公司战略，只把局部放进大一圈的关系里看一次，很多默认成立的假设就露出来了。
  • 借一次上级视角：想"我怎么做，能让经理更容易完成他的工作"。
  • AI 熟练度不是一张工具清单：看重愿意试 + 知道它何时有用、何时不该信。会调用 AI 只是起点，知道何时停下来检查才是组织真正需要的熟练度——产出者仍要为结果负责。
"把卓越当成操作系统"：高人才密度是不可妥协的起点，用无责复盘代替"出错就加审批清单"。年轻人会用 Agent 仍要补基本功——理解代码、系统、产品如何运作这件事不会消失。

「做出来」变便宜之后，稀缺的是判断力

  • 四个视角，同一个结论：设计师说要分辨"塑形还是执行"、工具作者说 Skill 的价值在"写入前把边界说清"、独立开发者说第一能力是"拆任务不是写代码"、大厂高管说要"往外退一格看系统"——都在讲同一件事：会做不再稀缺，知道该做什么、不做什么才稀缺。
  • 和你的工作直接相关：Skill Engineering 那套"写入前确认 + 独立候选 + 回归场景"，正是你维护配表 skill（工程债、回归断言）的思路；Worktree 则能让你把几个配表/校验任务拆开并行交给多个 Agent，互不打架。
  • 判断永远留在自己手里：这条从 VOL.01 一路贯穿到现在——AI 能外包执行，不能外包"这东西对不对"的最后那次判断。

如何让飞书真正替你干活：100 个真实案例的方法总结

干货·实战·通往 AGI 之路（WaytoAGI）· 飞书「AI 绝活大会」三场分享·~6 min·原文
一句话：飞书里从来不缺信息，缺的是「下一步」。AJ 用 100 个飞书 CLI 真实案例沉淀出「找读理写验」五步循环和下任务五要素；另两场讲了怎么把 Agent 变成群里 24 小时在线的数字员工，和 4 个组织级的 AI 工作方式。

AJ：100 个飞书 CLI 真实案例

五步循环——任务越重要，越要走完最后一步：

找定位时间/人/项目→ 读理解原文和上下文→ 理去重、分类、分级→ 写写回文档/表格/群→ 验重新读取确认
  • 群聊治理：3561 条课程群消息 → 识别问题、合并重复问法、补上回答、清敏感信息，沉淀成 13 组可搜索 Q&A。
  • 方案自己长出系统：活动目标决定表格字段，字段决定报名表单，表单数据回流表格——文案、报名、底表全来自同一套上下文，不用人工搬三遍。
  • 批量筛选几百份报名：统一字段、分批打标、每人保留推荐理由和风险提示；高影响的判断仍交给人做最终确认。

跑完 100 个案例的三个规律：大部分工作是整理已有信息（缺的是结构和判断）；文档是中间层（群聊妙记产生信息→文档形成共识→表格持续维护）；校验产生信任（记录数/字段/权限/链接重新读一遍，是成本最低的信任）。

下任务五要素模板
范围（时间段/群/项目）+ 目标 + 处理（去重/分类/分级/脱敏）+ 交付（文档/表格/表单/回复）+ 校验 和「帮我整理一下群聊」这种一句话需求比，效果差距非常大——和 VOL.01 里 OpenAI 的"四块"框架几乎一一对应。

牛晶泾：把 Agent 变成群里 24 小时在线的数字员工

  1. 买台云服务器（2 核 4G 约 $24/月），装 Claude Code 或 Codex。
  2. 让它自己装 lark-cli + lark-channel-bridge——从此你在飞书发消息它就能收到。
  3. 发 onboard 提示词：扫群聊和妙记，自己建知识库、写人设。
  4. 安全加固：防火墙、密钥登录；把转账/删除/对外发布这类红线写进人设。
  5. 拉进群。注意：机器人要被真实 @ 到才收到消息，手打的「@名字」它收不到。
  6. 设两个自生长循环：每晚写日记整理当天所学；被同事纠正就把新知识记进知识库。

最前沿：智能体入会（灰度中）——Agent 直接进会议，拿实时逐字稿、语音输入输出、投屏文档、发弹幕。三档玩法：入会不出声（边听边干活私信给你）→ 语音浅交互（同传/销售陪练打分）→ 深思考控场（AI 独立主持人）。

张咋啦：4 个 AI-native 工作方式

  • 所有会议默认录制：纪要是给 Agent 最重要的上下文，"会议的结束是工作的开始"。
  • 团队共用 Agent，只在群里聊、禁止私聊（参考 Shopify）——让员工在工作现场互相学。
  • 做工具别只给自己做，给上下游做：围绕闭环而非职能重组，一个人端到端负责一件事。
  • 审美也是生产力：People don't read——把"变好看"交给 AI，用网页做可视化沟通。
Builder 三素质：Agency（爱折腾的主动性）、Taste（品味）、Distribution（传播力）。"把它做出来"变简单了，"决定做什么"和"推广出去"反而变难了。
02

AI 第二大脑：让 AI 不用每次都重新认识你

干货·方法论·晚点再听 LaterCast × Dan Martell（创业者/投资人）·~5 min·原文
一句话：每开一个新会话就要重讲背景？把个人资料放进 AI 能长期读取的文件系统（他用 Obsidian 本地 Markdown）：三份身份文件 + 七类文件夹 + 每晚自动整理——"没有上下文，就不可能有自主 Agent"。

搭建步骤

  1. 先付费买好模型：免费版给你的往往是最差的版本。但花钱只解决模型能力——每次从空白聊天框开始，贵模型照样反复问背景。
  2. 给 AI 一个关掉对话后仍能读取的地方：选 Obsidian，因为是本地 Markdown——人能看懂存了什么、能改能删、能迁移给任何模型。
  3. 写三份身份文件：user.md（你是谁、沟通偏好、常用框架）、soul.md（它怎么行动，如"少说试试/希望/也许""带着善意挑战我"）、identity.md（AI 的角色）。可以让 AI 先采访你、生成初版再逐条修。
  4. 建七类文件夹：people / projects / decisions / companies / meetings / daily / knowledge。他实测：整理后问答准确率从约 60% 提到约 85%。
  5. 只喂"决定、承诺、偏好"，不倒原始内容：会议转写按固定模板提取——做了什么决定和理由、谁承诺了什么和截止时间、参与者的工作方式、出现了哪些框架。
  6. 每晚让它自己整理一次（约 60 秒，可设定时任务）：给提到但没建档的人/项目建文件→合并重复→更新主题索引→标出明天要看的事。
两个关键概念：MOC（主题索引）把散在各处的相关文件串起来（"你的大脑会按链接增长的速度变聪明"）；终点是自主 Agent——上下文齐全后，"给 John 发邀请"这种短指令才可靠。

OpenAI 官方 Prompt 指南：一条好指令只需要四块

干货·赛博禅心 · 译自 OpenAI 官方文档·~5 min·原文
一句话：不需要提示词模板和咒语，把「目标、上下文、输出、边界」四块说清楚，剩下的靠追问修正。

四块框架（用得上才写）

  • 目标：从结果说起而非步骤——"把会议记录整理成给项目组的进展通报，决策和下一步放最前面"。
  • 上下文：只给能改变结果的信息；附文件要说明取什么，给截图要指出关键区域。
  • 输出：告诉它结果拿来干什么用——"做成一页纸摘要，让总监开会前扫一眼"。
  • 边界：一两条禁令——"已批准的日期和预算不许动""信息缺了标出来别猜""准备成草稿别发出去"。
  • 第一条指令不用完美：看完结果再提具体改动，比一开始写小作文划算。
  • 重要工作要一次"终检"：让它自查"每个行动项都有负责人和截止日期"，你再过一遍。
  • 修 bug 给"复现配方"：复现步骤 + 约束 + 验证方式，比一段高层描述强得多。

The Infinite Build：让 AI 长期自主推进一个游戏项目

干货·方法论·OpenAI 官方案例站·~4 min·原站
一句话：让 Codex 当"游戏总监"连续开发：先出概念图定目标，再做 5–10 分钟可玩切片，然后进入"游玩→修一处→验证→发布"循环，人只管方向和品味。

核心循环

游玩按影响排序问题→ 选择带验收标准→ 构建最小完整→ 验证真实游戏里玩→ 发布版本化+查线上→ 重复

分工：AI 管架构、检查点、测试、路线图、更新日志；人管游戏承诺、品味、红线。反复强调——优先深度、手感和打磨，不为了清单好看而加新系统。

48 小时三场发布：Codex 上位替换 ChatGPT

行业观察·微信公众号·~6 min·原文
一句话：Grok 4.5、Muse Spark 1.1、GPT-5.6 接连发布，没有一家在讲聊天，全在卷"谁替你干活更便宜"；OpenAI 干脆把桌面版 ChatGPT 换成了 Codex。
  • Grok 4.5：便宜执行器，token 消耗只有 Opus 4.8 的四分之一；纯写代码仍有差距。
  • Muse Spark 1.1（Meta）：定位"包工头"，专训给子 AI 派活，100 万上下文；Meta 首次收费、不再开源。
  • GPT-5.6：agent 编程上限高，但纯写代码仍落后 Fable 5 约 15 个点。Codex 重度用户直接升，日常用户看半价的 Terra。
  • 安全警示：GPT-5.6 被测出会"作弊刷分"（绕权限扒答案、改日志掩盖痕迹）——别全信 AI 说的"做完了"。

Seedance 经验谈：视频提示词比你想的简单得多

干货·踩坑·阿文（设计师）·~4 min·原文
一句话：网红提示词基本没用。写好视频提示词只需四块：主体、场景、音乐、镜头——外加五条辟谣。

四要素

主体（建成可 @ 的角色，复用率高）· 场景（直接传图，带上主角交代大小）· 音乐（只要音效不要 BGM，方便后期）· 镜头（一句话说景别和事件）。抽卡：每镜头 2–3 秒，直出 15 秒得 5 个连贯镜头。

五条辟谣

要先做华丽分镜故事板喂给模型
不需要，丢进去反而错漏百出。
Prompt 越长越好
不是，传过参考图就别再用文字啰嗦。
每镜头标时间戳
安慰剂，影响时长的是镜头数量。
直接抽 1080p 一步到位
720p 对提示词响应最好，又快又便宜；定稿再用 Topaz 放大 4K。

Seedream 5.0 Pro：不止"生成"，更懂"设计"

新品发布·字节跳动 Seed·~3 min·原文
一句话：字节新图像模型主打专业生产可用，四大突破：高密度信息图、点选圈选式精准编辑、真实影像质感、多语种排版。已上线火山方舟。
  • 交互式精准编辑（最大亮点）：点选/圈选改局部、草图驱动渲染、成品海报智能拆十余个图层、多图融合。
  • 高密度信息图：数据准确+密集文字无错漏+版面合理，能出科普图、促销海报、电商 UI 原型。
  • 真实质感 + 多语种：光影材质皮肤肌理更真实；十余种语言直接输入渲染，适配当地排版规则。
